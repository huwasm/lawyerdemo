## PROJECT OVERVIEW

**What to build:** A Next.js web app that automates personal injury law firm intake. A team member uploads a police report PDF, AI extracts data, auto-matches to a client in Clio Manage (legal CRM), team verifies, presses approve, and everything else happens automatically: Clio fields updated, retainer agreement generated, calendar deadline created, personalized email sent to client.

**Tech stack:**
- Frontend: Next.js (React) + Tailwind CSS
- Backend: Next.js API routes
- AI: Claude Vision API (for PDF extraction) + Claude API (for email drafting)
- CRM: Clio Manage REST API v4
- Email: Resend API
- Deploy: Vercel
- Color scheme: Clio brand (#0070E0 blue, #333333 dark, #FFFFFF white)

**Reference file:** `dashboard-demo.html` shows the UI mockup with Clio colors and the full interaction flow (upload → extract → verify → approve → processing steps → success).

## CRITICAL RULES

1. **Client is NOT always Vehicle 1.** In 2 of 5 test reports, the client is Vehicle 2 (pedestrian or bicyclist). The AI must extract ALL names. The system matches against the Clio Matter's client name to identify who is client vs defendant.

2. **Police report is neutral.** It doesn't say who the client is. Only the Clio Matter knows.

3. **Registration plate may be empty.** If client is a pedestrian or cyclist, there's no plate. Use "N/A" as fallback.

4. **Retainer MUST be generated by Clio's document automation** — not generated externally. Use `POST /document_automations`.

5. **Email is NOT sent through Clio.** Our automation sends it (Resend).

6. **The filename of the uploaded PDF is meaningless.** Could be "scan003.pdf" or "aslkdfj.pdf". Never use it for matching.

## AI EXTRACTION PROMPT

The extraction prompt lives in `lib/providers/types.ts` (single source of truth). It instructs the AI to return structured JSON from the MV-104AN form. Key points:

- Extracts ALL names from both vehicles (V1 + V2), plus pedestrians/cyclists
- Vehicle 2 may be PEDESTRIAN, BICYCLIST, or OTHER PEDESTRIAN — the AI checks the form checkboxes
- `ins_code` is the insurance company code on the form (e.g. "0042")
- `is_other_pedestrian` covers the "OTHER PEDESTRIAN" checkbox on the form
- If pedestrian/bicyclist, vehicle info (plate, make, year, ins_code) will be empty strings
- Confidence scores (0-100) are self-reported by AI for 7 key fields

### VehicleInfo fields extracted per vehicle:
```
driver_name_first, driver_name_last, sex, date_of_birth,
address, city, state, zip,
plate_number, plate_state, vehicle_year, vehicle_make, vehicle_type, ins_code,
is_pedestrian, is_bicyclist, is_other_pedestrian
```

### Top-level fields:
```
accident_date (MM/DD/YYYY), accident_time (HH:MM),
no_injured, no_killed, no_vehicles,
accident_location { road, intersecting_street, borough },
vehicle_1, vehicle_2,
officer_notes, all_persons_involved,
confidence
```

**Important:** The AI must handle that Vehicle 2 can be a PEDESTRIAN, BICYCLIST, or OTHER PEDESTRIAN (checked at the top of the form). In those cases, vehicle info will be empty.

## RETAINER TEMPLATE MERGE FIELDS

The retainer agreement template (`Retainer Agreement - Richards & Law [Hackathon].docx`) uses these merge fields in `[square brackets]`:

| Merge Field in Template | Source |
|---|---|
| `[Client Name]` | Matched client from police report |
| `[Law Firm Name]` | Hardcoded: "Richards & Law" |
| `[Date of Accident]` | From police report |
| `[Defendant Name]` | The other party from police report |
| `[his/her]` | Client sex: M→"his", F→"her" |
| `[he/she]` | Client sex: M→"he", F→"she" |
| `[Accident Location]` | From police report (road + intersection + borough) |
| `[Registration Plate Number of the Client's Car]` | From police report (or "N/A" if pedestrian/cyclist) |
| `[Statute of Limitations Date]` | Calculated: accident date + 8 years |
| `[Name of Client]` | Same as Client Name (signature line) |
| `[Name of Attorney]` | Hardcoded: "Andrew Richards" |

**Conditional paragraphs (in the Word template):**
- IF No. Injured > 0 → include bodily injury paragraph
- IF No. Injured = 0 → include property damage only paragraph

These conditions are handled by Clio Draft's conditional logic in the Word template, driven by a custom field value.

## KNOWN ISSUES

1. **Contact email retrieval** — `GET /contacts/{id}?fields=email_addresses` returns IDs but not the actual address. Nested field syntax `email_addresses{id,address}` failed. Fallback: uses `HACKATHON_EMAIL` env var.

2. **PDF download from Clio** — `GET /documents/{id}/download` returned empty. Try `GET /document_versions/{version_id}/download`. May return binary or a redirect URL. Email still sends even if download fails (retainer step is non-blocking).

3. **US Clio account** — Currently testing on EU (`eu.app.clio.com`). For final submission, need a US account (`app.clio.com`). APIs are identical, only base URL changes.

4. **OAuth token refresh** — Current token may expire. Need to implement token refresh using `grant_type=refresh_token` or re-authorize.

5. **Clio updateMatterCustomFields** — When existing custom field values are present on a Matter, the PATCH may fail or overwrite. Need to test with pre-populated fields.

6. **Number Injured extraction** — Vincent case returned `no_injured=2`, ANALYSIS.md expects 0. Needs prompt tuning or post-extraction verification.
